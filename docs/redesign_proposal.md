# 再設計提案: スケッチ・線画の姿勢推定と構図修正 (2026-07-03)

現行アプローチ (RTMPose-m + Bizarre Pose 3.2K + 教師信号改善) の限界を認めた上で、
「イラストのラフ・線画に対して姿勢を推定し、構図の誤りを指摘・修正提案する」という
最終目的から逆算した再設計。既存資産の温存は前提にしない。

## 1. 現行アプローチの限界の構造化

autoexp 29実験 + Phase 0-2 の結果から、限界は3層に分解できる:

### 1-1. データの壁 (最も深刻)
- 学習データはイラスト 3,200枚のみ。backbone 強化 (HRNet/ViTPose/UniFormer)、
  Amateur Drawings 178K 追加、すべて失敗 → **この規模ではモデル容量もデータ量も効かない**
  ことが実証済み
- v=1/v=0 のアノテーションは手修正頼み。occlusion / extreme / chibi の教師データは
  実質存在しない (mydata 拡充計画も「評価用」で 60枚どまり)
- 線画・ラフというターゲットドメインの学習データは **0枚** (BP は完成イラスト)

### 1-2. 表現の壁 (構図修正には COCO17 2D keypoint が根本的に不足)
- 構図の誤りとは: 手足の長さの矛盾、パースと肢体の短縮 (foreshortening) の不整合、
  頭身と体格の不一致、関節可動域の逸脱、接地・接触の破綻。
  **これらは 2D 座標17点からは原理的に判定できない** (短縮なのか短いのか区別不能)
- chibi で relation prior が壊れるのは「頭身」という変数をモデルが持たないから。
  2D keypoint 表現のままでは頭身は永遠に外乱でしかない
- 「修正」を出力するには、正しい姿勢空間 (prior) からの逸脱量と修正方向が必要。
  2D 座標にはその文法がない

### 1-3. 精度の壁 (現行路線の残り伸びしろは小さい)
- what-if 上限: miss 全修正 +0.085 (@50) / jitter 全修正 +0.091 (@75)。
  どちらも究極的にはデータ量の問題であり、1-1 に帰着する

**結論: モデルの改良ではなく、(a) データ生成方法と (b) 姿勢の表現形式を変える必要がある。**

## 2. 提案アーキテクチャ (3層パイプライン)

```
入力: ラフ/線画/完成イラスト
  │
  ├─ Layer 1: 知覚 ────────────────────────────────
  │   2D keypoint + per-keypoint presence/visibility (現行資産の後継)
  │   + 頭身・体型パラメータ推定
  │   + 3D pose lift (SMPL 系 + プロポーション変数)
  │   学習データ: 3D モデル大量レンダリングによる合成データ (§3)
  │
  ├─ Layer 2: 診断 ────────────────────────────────
  │   3D 表現上でのルール/prior ベース検査:
  │   - 左右肢長の矛盾 (短縮で説明できない差分)
  │   - 関節可動域逸脱、体幹の捻れ限界
  │   - 頭身に対する手足サイズの不一致
  │   - パース (カメラ) と各肢の短縮方向の整合性
  │   - 接地・接触の破綻
  │   → 「どこが・なぜ・どの程度おかしいか」の構造化出力
  │
  └─ Layer 3: 修正提案 ─────────────────────────────
      診断結果を pose prior (VAE latent) 上で最近傍の妥当姿勢へ射影
      → (a) 骨格オーバーレイ表示 (軽量・確実)
      → (b) 修正骨格を条件にした pose-conditioned 再描画ガイド
           (ControlNet + アニメ系拡散モデル、オプション)
```

ポイント: **修正は 3D で行い、表示は 2D に戻す**。2D のまま「正しい位置」を
指すことは不可能 (正解が多義的) だが、3D なら「この肘はこの角度が限界」と言える。

## 3. 柱1: データ戦略 — 層で分ける2段構え (データの壁の突破)

### 3D 合成データへの疑義 (2026-07-03 議論で確定した設計判断)

「3D レンダリングはスケッチの特徴分布を生成できないのでは」という疑義は正当で、
**層によって答えが分かれる**:

```
ピクセル → 2D骨格      : スタイル依存。3D合成の sim2real ギャップが直撃する層
2D骨格 → 3D・診断・prior : スタイル不変。3D合成が正しい道具である層
```

- 手描きラフの特徴 (線の強弱・重ね線・アタリ線・意図的誇張・2D的見栄え優先で
  3D非整合なポーズ) は「3Dシーンの投影+輪郭線抽出」では作れない。
  **絵はレンダリングの逆写像ではない**。3Dレンダは常に幾何整合した clean な線を
  出すため、ピクセル層の学習は「整った線」に過適合するリスクが高い
- 傍証: Stage C (Amateur Drawings 178K) の分布ミスマッチ失敗。
  BP 著者らが合成でなく実イラスト 4K を構築した事実 (合成だけでは届かなかった)
- 一方 Sketch2PoseNet の learn-from-synthesis が成立するのは
  **入力が骨格という抽象表現でありスタイルが存在しない**から。
  3D→2D投影+肢長摂動という合成は骨格レベルの操作で、ギャップが原理的に生じない
- ピクセル層での解決策は 3D の放棄ではなく**変換器の差し替え**: 手続き的
  輪郭線抽出をやめ、実イラストで学習済みの拡散モデルを「3D→スケッチ」の
  レンダラにする (§3a)。手描き統計は生成 prior から、幾何とラベルは 3D から

### 3a. ピクセル層 (画像→2D骨格): 3D→スケッチ生成合成が主力 (2026-07-03 改訂)

方針: **3D は幾何とラベルの足場、手描き統計は実イラストで学習済みの生成 prior が
担う**。§冒頭の疑義は「手続き的レンダリング (Freestyle 等の輪郭線抽出) では
手描きの分布を作れない」ことへの反対であり、変換器を学習済み拡散モデルに
替えることで解消する。同時に、ポーズが 3D サンプラー由来になるため
「BP 3.2K が解けるポーズしか教えられない」というスタイル変換・擬似ラベルの
限界 (循環問題) も超える — **スタイルとポーズ多様性の両ボトルネックを同時に攻める**。

パイプライン:

```
① 幾何生成:   VRoid素体(頭身2-8) × ポーズ(AMASS/MMD+摂動) × カメラ(見切れ・俯瞰煽り)
              → レンダーパス: toon / 輪郭線 / depth + 正解 (2D/3D kp, v=0/1/2, 頭身)
② スタイル変換: 輪郭線レンダを条件に ControlNet(lineart/depth) + anime系SD img2img。
              ラフ・色鉛筆等のスタイルは Danbooru sketch / traditional_media タグで
              学習した style LoRA で指定 (mydata は使わない — ベンチマーク保護)
③ アナログ化:  紙テクスチャ合成・撮影シミュレーション (パース歪み・照明ムラ・机背景)。
              手続き処理で十分な層
④ 品質ゲート:  生成画像を再推定し条件骨格と乖離したサンプルを棄却。
              「安定して合成」は生成品質でなくフィルタで保証する。
              合格率 vs スタイル強度 (denoise strength) が最初の探索パラメータ
```

- chibi は「chibi 頭身の 3D 素体をレンダした輪郭線」を条件にするため、
  人体比率前提の openpose 条件より頭身が保たれる
- v=0 (見切れ)・極端カメラの正解座標が体系的に作れる (実データでは不可能な教師)

補助データ (生成合成と混合):

1. **実データ×スタイル変換**: BP 3.2K を線画・スケッチ風変換 (幾何不変で
   ラベル移植可)。本物の絵描きのポーズ分布を保つアンカーとして併用
2. **擬似ラベル**: S2/P3 + presence フィルタで Danbooru sketch/lineart タグ画像へ。
   新規ポーズは学べない (循環) ため、スタイル頑健化のみ期待
3. **能動学習アノテーション (フォールバック)**: 生成合成が数値で出なければ、
   失敗モード (wrist/elbow・chibi) に絞った実イラスト人手ラベルが最終手段

### 3b. 骨格層 (2D骨格→3D・prior・診断): 3D合成が主力

- 素体: VRoid Studio パラメトリック生成 (頭身 2〜8 連続スイープ) — ここは
  レンダ画質が無関係なのでライセンス・画風偏りの問題も軽い
- ポーズ: AMASS / Mixamo / MMD からサンプリング + 関節角摂動 + 肢長摂動
  (Sketch2PoseNet 同様「絵の誇張」を骨格レベルで模倣)
- 教師: 2D投影骨格 → 3D姿勢・頭身・visibility (v=0/1/2)・短縮量。すべて完全精度
- pose prior (VAE) の学習データもここから

### 検証方法 (最小コストで仮説を切る)
具体的な技術スタック・手順・判定基準は **`docs/r2a_investigation.md`** に分離
(2026-07-03 調査済み)。R2 を2段に分ける:
- **R2a (生成品質の検証)**: 3D→スケッチ変換の少量サンプル (数百枚) を作り、
  (a) スタイル距離 (Danbooru sketch タグ実画像との FID/CLIP 距離)、
  (b) ラベル忠実度 (品質ゲート通過率、目視スポットチェック) を測る。
  ここで「安定して合成可能」かが決まる。**学習より先にデータだけで判定**
- **R2b (学習効果の検証)**: R2a 合格後、数万枚生成 + BP 3.2K 混合で
  現行 RTMPose-m を再学習 → mydata lineart/chibi と Crop-Bizarre で S2/P3 比較

## 4. 柱2: 3D 表現への移行 (表現の壁の突破)

### 基盤: Sketch2Pose 系
- [Sketch2Pose (SIGGRAPH 2022)](https://dl.acm.org/doi/abs/10.1145/3528223.3530106):
  ビットマップスケッチ1枚から SMPL 3D 姿勢を推定。骨の接線・自己接触・短縮を
  予測して多義性を解消。**コード公開済み** (github.com/kbrodt/sketch2pose)
- [Sketch2PoseNet (SIGGRAPH Asia 2025)](https://dl.acm.org/doi/10.1145/3757377.3763855):
  最適化不要の feed-forward 版。VAE pose latent + 肢長摂動 augmentation
- まず **既製 Sketch2Pose を mydata に流して素の性能を測る** (実装コストほぼゼロ)。
  人体プロポーション前提なので chibi で壊れるはず → そこが我々の差分価値

### アニメ向け拡張 (研究的コア・差別化ポイント)
- SMPL に **頭身スケールパラメータ** を追加した変形体型モデル
  (骨長を頭身関数でスケール; VRoid のパラメトリック素体と対応させ、
  合成データ側で頭身ラベルが無料で手に入るので教師あり学習が可能)
- これにより「chibi は誤差」ではなく「頭身 2.5 の正しい姿勢」と表現でき、
  診断層は「その頭身にしては腕が長い」を言えるようになる

### 修正提案の生成
- 合成データのポーズ分布で VAE prior を学習 (Sketch2PoseNet と同じ道具立て)
- 診断で異常とされた姿勢を latent 上で最近傍の妥当姿勢に射影 → 修正案
- 2D 投影して骨格オーバーレイ / ControlNet (openpose + lineart 条件) で
  ユーザーのタッチを保った再描画例を提示

## 5. 柱3 (補助): VLM を診断・講評レイヤーに使う

Qwen2.5-VL 等の VLM は LoRA fine-tune で keypoint 検出も可能と報告されているが
([surgical tools での実証例](https://arxiv.org/html/2508.20830v1))、
精密な座標回帰は専用モデルに劣る。適所は Layer 2/3 の言語化:

- 入力: 画像 + Layer 1/2 の構造化出力 (keypoint, 頭身, 診断フラグ)
- 出力: 「右腕が肩から不自然に長い。この俯瞰なら前腕は短縮して見えるはず」
  という **絵描きに通じる講評文**
- keypoint エンジンとしての VLM 置き換えは、専用モデル路線が数値で行き詰まった
  場合の代替案として保留 (LoRA 実験自体は安価なので並走可)

## 7. フェーズ計画

| Phase | 内容 | 判定基準 | 期間目安 |
|-------|------|---------|---------|
| R1 | 既製 Sketch2Pose を mydata/BP に適用、素の性能と破綻箇所を測定 | chibi/線画での破綻を定量化 | 数日 |
| R2a | 3D→スケッチ生成合成のサンプル生成 (数百枚) と品質測定 | スタイル距離 + 品質ゲート通過率 (学習前判定) | 1-2週 |
| R2b | 数万枚生成 + BP 混合で RTMPose-m 再学習 | mydata lineart/chibi OKS@50 が S2 比 +0.05 以上 | 2-3週 |
| R3 | 規模拡大 (10^5) + ViTPose 級へ移行、頭身推定を追加。実データスタイル変換・擬似ラベルも混合 ablation | BP test ≥ S2 かつ chibi/extreme で大幅改善 | 3-4週 |
| R4 | 3D lift (頭身付き SMPL) + 診断ルール実装 | mydata に人工エラーを注入した診断ベンチで precision/recall | 4週〜 |
| R5 | 修正提案 UX (オーバーレイ → ControlNet 再描画) | ユーザー評価 | — |

R1/R2 が最初の意思決定点。**R2 で合成→実イラストの転移が確認できなければ
スタイルギャップが本丸なので、レンダリング多様化に投資を集中する**
(モデル側をいじらない — これまでの教訓)。

## 8. リスク

1. **幾何ドリフト**: 拡散モデルは高 denoise で肢を描き直す (ラベルと画像が乖離)。
   低 denoise では手続きレンダに近い「整った線」に戻る。→ 品質ゲート必須 +
   合格率とスタイル強度のトレードオフを R2a で最初に探索。
   ゲート自体が現行モデル依存 (難サンプルほど誤棄却) というバイアスにも注意
2. **スタイルのモード崩壊**: style LoRA が特定画風に偏ると多様性が死ぬ。
   LoRA を複数系統 (デジタルラフ / 色鉛筆 / ペン入れ線画) 用意しスタイル距離で監視
3. **mydata 汚染**: style LoRA 学習・生成調整に mydata を使うとベンチマークが死ぬ。
   スタイル参照は Danbooru sketch / traditional_media タグに限定し、
   mydata は最終評価のみ (mydata_expansion.md の規約)
4. **ライセンス**: VRoid Hub モデルは個別ライセンス。学習利用可のセット
   (自作パラメトリック生成 / 許諾済みセット) に限定する運用が必要
4. **3D 化のコスト**: R4 は研究要素が強い。R2/R3 (2D のまま合成データで強化) だけでも
   線画・chibi の推定精度は上がるので、段階的に価値が出る構成にしてある
